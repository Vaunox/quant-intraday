## 0. Goal and scope

**Goal:** stress the registered ensemble + regime-gate strategy from the five §4b.7 angles and
reconcile a sample strategy across two independent backtest engines — the inputs to the kill-gate's
criterion 6 (*"survives the robustness battery"*), emitted in P2.9.

**What the battery does (all model-agnostic primitives in `research/validation/robustness.py`;
the model is injected by `research/pipeline/robustness.py`):**

1. **Parameter sensitivity** — re-evaluate the strategy under small single-knob perturbations
   (blend method, regime-gate on/off, regime count, CPCV embargo). A robust edge's median
   path-Sharpe barely moves; a **knife-edge** swings wildly or flips sign (overfit).
2. **Monte Carlo trade shuffle** — randomise the order of the realised per-trade returns and
   measure the equity path's max drawdown. If the observed ordering is anomalously benign, the
   smooth equity curve was lucky sequencing, not a robust edge.
3. **Noise injection** — add Gaussian noise (scaled per feature std) and watch the edge decay. A
   real edge degrades gracefully; an overfit one collapses.
4. **Cross-symbol validation** — leave-one-symbol-out: train on the rest, score the held-out name.
   A universal signal generalises; a symbol-specific one overfits out of universe.
5. **Synthetic-data backtest** — run the *same* recipe on driftless GBM price paths that contain
   **no real historical luck**. The edge must centre near zero; a strong edge there is a
   methodology leak.

**Two-engine reconciliation** (`research/validation/reconcile.py`): the artifact's own signal is
driven through both the P2.1 event-driven `Backtester` and an independent, fully **vectorised**
`VectorizedBacktester`. They share the injected Indian cost + slippage models but differ entirely
in simulation structure, so agreement within tolerance cross-validates the error-prone accounting
(signed notional, cost application, square-off timing, mark-to-market).

> **Design note (why an in-house second engine, not VectorBT/Backtrader).** §4b.8 names those
> libraries as *examples* of the fast-vectorised-vs-event-driven pair, but equally sanctions
> implementing the math directly. The whole codebase hand-rolls over heavy deps to keep the engine
> env lean, and a third-party engine knows nothing of our Indian cost model or next-bar-open +
> intraday-square-off rules — so it could never reconcile *within tolerance* without re-implementing
> those semantics inside it anyway. The independent vectorised engine is the simpler, more robust,
> more testable choice and is the one that can actually reconcile to floating-point noise.

**What "done" means (acceptance criteria):** each stress test runs and reports; the two engines
reconcile within tolerance on the sample strategy; persistent MLflow records every variant tried
(the honest trial count for P2.9's DSR); the run-IDs are recorded in `docs/PROGRESS.md`.

## 2. Run it (local)

Persistent MLflow, logging straight to the sqlite store (no background server — the engine-env
rule, see `docs/` memory note). Run from the repo root in the research env:

```bash
# Windows (PowerShell):
$env:MLFLOW_TRACKING_URI = "sqlite:///mlruns/mlflow.db"
$env:PYTHONUTF8 = "1"   # MLflow prints an emoji that crashes the cp1252 console without it
.venv-research\Scripts\python.exe scripts\run_robustness.py `
    --start 2021-06-24 --end 2026-06-23 --tracker mlflow --experiment p2.8-robustness

# Linux / macOS:
export MLFLOW_TRACKING_URI="sqlite:///mlruns/mlflow.db"
.venv-research/bin/python scripts/run_robustness.py \
    --start 2021-06-24 --end 2026-06-23 --tracker mlflow --experiment p2.8-robustness
```

`--symbols` defaults to the configured universe; `--registry-dir` to the pipeline config. A **fast
smoke** shrinks the battery's knobs without touching code (Ground Rule 2):

```bash
QUANT__robustness__cpcv_groups=4 QUANT__robustness__cpcv_test_groups=1 \
QUANT__robustness__noise_repeats=1 QUANT__robustness__monte_carlo_shuffles=200 \
QUANT__robustness__synthetic_universes=2 QUANT__robustness__synthetic_sessions=15 \
QUANT__model__num_boost_round=60 \
.venv-research/bin/python scripts/run_robustness.py --start 2021-06-24 --end 2026-06-23 \
    --tracker mlflow --experiment p2.8-robustness
```

**Exit code:** `0` if every test ran and the engines reconciled; `1` only if the engines disagree
(an implementation bug to fix before any capital rides on the backtest). **A weak edge is not a
failure** — the battery reports it, and the kill-gate verdict is P2.9's to emit.

## 3. Read the output

Each test logs an INFO event and an MLflow run. The consolidated verdict line
(`robustness battery PASSED/FAILED`) reports: `parameter_sensitivity_knife_edge`,
`noise_edge_persists`, `monte_carlo_percentile_rank`, `cross_symbol_generalizes`,
`synthetic_edge_detected`, and `reconcile_max_money_diff`. List the MLflow runs with:

```bash
.venv-research/bin/python - <<'PY'
import mlflow
from mlflow.tracking import MlflowClient
c = MlflowClient()
exp = c.get_experiment_by_name("p2.8-robustness")
print("experiment_id", exp.experiment_id)
for r in c.search_runs([exp.experiment_id]):
    print(r.info.run_id, r.data.tags.get("mlflow.runName"), r.data.metrics)
PY
```

**Honest reading (Inviolable Rule 7).** On a weak edge most flags will be unfavourable — that is
the expected reality, not a bug. The battery's job is to *measure and report*; whether the edge
clears the seven-point kill-gate is P2.9's verdict.
