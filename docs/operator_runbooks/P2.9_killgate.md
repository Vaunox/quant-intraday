# P2.9 — Validation report + seven-point kill-gate (operator walkthrough)

**Subtask:** P2.9 (Phase 2 — Research Layer). See the master blueprint, Part IV (P2.9), Part III
**Layer 2 kill-gate**, and Deep Dive #2 (**"The kill-gate"** + §4b.8 output contract).
**Depends on:** P2.2 (CPCV/DSR/PBO) and P2.8 (robustness battery) — and a registered artifact.
**Audience:** the operator. The run reads the local Parquet archive + the registered artifact and
writes only an MLflow record + a rendered report; no money, no orders. The AI can execute and
verify it; this doc records the recipe.

---

## 0. Goal and scope

**Goal:** one report that decides **trade / don't-trade**. It assembles every piece of evidence
(Deep Dive #2 §4b.8 output contract) and emits the **seven-point kill-gate** as a PASS/KILL
verdict:

1. CPCV median path-Sharpe > 1.0, **annualised, after full Indian costs + slippage**.
2. Deflated Sharpe Ratio positive & significant given the honest trial count.
3. PBO below the threshold.
4. Narrow, positive CPCV path distribution (few/no negative paths, none deeply negative).
5. P&L not concentrated (profit factor + single-trade share).
6. Survives the robustness battery (P2.8).
7. Edge stable across regimes.

The thresholds are fixed in `config/default.yaml` under `kill_gate:` **before** the run (Inviolable
Rule 1 — no moving the goalposts to make a candidate pass).

**The discipline that makes it honest: every number is net of costs and point-in-time.** Criterion
1 is *"after costs"*, so the CPCV returns are netted by a representative round-trip cost (the Indian
cost model + slippage, charged once per triple-barrier event) and annualised by the data-derived
events-per-year. The **gross** ~0.04 the robustness battery reports is *not* the kill-gate input.

**What "done" means:** the report generates end-to-end and emits the verdict; persistent MLflow
records the run (the honest trial count for the DSR); the verdict is recorded in the iteration log.

## 2. Run it

From the repo root, in the research env (persistent MLflow → sqlite, no server):

```bash
# Linux / macOS:
export MLFLOW_TRACKING_URI="sqlite:///mlruns/mlflow.db"
.venv-research/bin/python scripts/run_validation_report.py \
    --start 2021-06-24 --end 2026-06-23 --tracker mlflow --experiment p2.9-killgate \
    --output docs/iteration_log/cycle-N_verdict.txt

# Windows (PowerShell):
$env:MLFLOW_TRACKING_URI = "sqlite:///mlruns/mlflow.db"; $env:PYTHONUTF8 = "1"
.venv-research\Scripts\python.exe scripts\run_validation_report.py `
    --start 2021-06-24 --end 2026-06-23 --tracker mlflow --experiment p2.9-killgate `
    --output docs\iteration_log\cycle-N_verdict.txt
```

A **reduced-knob smoke** (sufficient to confirm a known-weak candidate's KILL without the deferred
full-fidelity cloud spend — Part II iteration discipline) shrinks the battery + boosting via env
overrides, e.g. `QUANT__model__num_boost_round=80 QUANT__robustness__monte_carlo_shuffles=200
QUANT__robustness__synthetic_universes=2`.

**Exit code:** `0` once the report generates and the verdict is emitted. **A KILL is a successful
report, not an error** (Inviolable Rule 7) — only an exception (no data, no artifact) exits non-zero.

> **Cost note.** The full-fidelity report (production `num_boost_round`, full battery, 8 symbols, 5
> years) is heavy. Per Part II's iteration discipline, **do not** spend the full cloud run merely to
> confirm a candidate the smoke already shows failing — that is a poor use of credits. The
> full-fidelity report + cloud earn their spend when a candidate's smoke path-Sharpe is meaningfully
> above zero. A reduced-knob local run is the right tool for a known-failing candidate.

## 3. Read the output + record the verdict

The rendered report (`--output`, and the INFO log) shows each criterion PASS/FAIL with its
observed-vs-threshold value and the overall verdict. Record it in the iteration log:

- **PASS** → route to Phase 3 (a passing `cycle-N_verdict.md` is the Phase-3 prerequisite).
- **KILL** → route back to Phase 2R (the diagnostic → one-input-family iteration → re-run P2.9).

List the MLflow run:

```bash
.venv-research/bin/python - <<'PY'
import mlflow
from mlflow.tracking import MlflowClient
c = MlflowClient()
exp = c.get_experiment_by_name("p2.9-killgate")
for r in c.search_runs([exp.experiment_id]):
    print(r.info.run_id, r.data.tags.get("verdict"), r.data.metrics)
PY
```

Append the verdict (the rendered block + the MLflow run-id + a one-line routing decision) to
`docs/iteration_log/cycle-N_verdict.md`. That file is the P2R.3 deliverable and, for a PASS, the
gate Phase 3 re-checks at every session start.
